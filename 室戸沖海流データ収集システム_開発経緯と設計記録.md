## 概要

### 目的

高知県室戸岬沖の海流（流速・流向）・水温・塩分データを毎日自動収集し、ブラウザ上のダッシュボードで可視化するシステムを構築する。釣り・漁業・海洋レジャーに関わるユーザーが、室戸沖の海況をリアルタイムに把握できるようにすることが最終目標。

### 対象ユーザー

- 室戸沖で漁業・釣りを行う個人（開発者本人を含む）
- スマートフォンからの閲覧が主要ユースケース
- 技術知識はなくても操作できるUI

### 解決したい課題

1. 室戸沖の海流情報は公的機関のWebサイトに分散しており、1画面で時系列比較できるツールがない
2. 既存の海洋データサービス（CMEMS等）はAPIベースで一般ユーザーが直接利用するには敷居が高い
3. 日次で更新されるデータを手動収集するのは継続不可能

## 技術選定と理由

### データソース：CMEMS（Copernicus Marine Service）

| 候補 | 評価 | 採否 |
|------|------|------|
| CMEMS GLORYS12 | 解像度1/12°（約9km）、日次、全球、無料、Python SDK完備 | **採用** |
| JCOPE2M（JAMSTEC） | 高解像度だが利用申請が必要、OPeNDAP/FTP運用 | 併存（フォールバック） |
| 気象庁 海流推定図 | 画像ベース、API非公開 | 不採用 |
| HYCOM | 解像度は同等だが日本近海の精度でGLORYS12が優る | 不採用 |

**採用理由：** CMEMS GLORYS12は無料・高品質・Python SDKが整備されており、個人プロジェクトに最適。2022年以降のデータが確実に利用可能。

### フロントエンド：Chart.js v4.4.0 + 静的HTML

| 候補 | 評価 | 採否 |
|------|------|------|
| Chart.js | 軽量（167KB）、CDN配信、学習コスト低 | **採用** |
| D3.js | 高機能だがオーバースペック、学習コスト高 | 不採用 |
| Plotly.js | 高機能だがファイルサイズ大（3MB+） | 不採用 |
| React + Recharts | ビルド環境が必要、保守負荷大 | 不採用 |

**採用理由：** GitHub Pagesでの静的ホスティングを前提とし、ビルドツール不要・CDN読み込みのみで完結する構成。保守者の技術レベルを問わない。

### 地図表示：Leaflet.js v1.9.4

| 候補 | 評価 | 採否 |
|------|------|------|
| Leaflet.js | 軽量（40KB）、CDN配信、衛星画像タイル対応 | **採用** |
| Google Maps API | APIキー必要、課金リスク | 不採用 |
| 静的画像 | ズーム不可、更新困難 | 不採用 |

### ホスティング：GitHub Pages

| 候補 | 評価 | 採否 |
|------|------|------|
| GitHub Pages | 無料、HTTPS自動、Actionsと統合 | **採用** |
| Netlify | 無料枠あるがGitHub完結にならない | 不採用 |
| 自前サーバー | 費用・保守コスト大 | 不採用 |

### 自動化：GitHub Actions

| 候補 | 評価 | 採否 |
|------|------|------|
| GitHub Actions | 無料（月2,000分）、リポジトリ直結 | **採用** |
| Coworkスケジューラー | PC起動必須、安定性に欠ける | Phase1まで暫定使用 |
| cron + VPS | 費用発生 | 不採用 |

## 課題と対応

### 課題1：`copernicusmarine.login()` のTypeError

- **発生状況：** GitHub ActionsでPythonスクリプト実行時
- **エラー：** `overwrite_configuration_file` パラメータに関するTypeError
- **原因：** `copernicusmarine` ライブラリのバージョン差異。GitHub Actions環境の最新版で `login()` のAPIが変わった
- **解決：** `login()` 呼び出しを削除し、環境変数 `COPERNICUSMARINE_SERVICE_USERNAME` / `COPERNICUSMARINE_SERVICE_PASSWORD` で認証する方式に変更。copernicusmarine SDKは環境変数を自動認識する
- **代替案：** `login(overwrite_configuration_file=True)` を試す案もあったが、バージョン依存のため環境変数方式の方が安定

### 課題2：スマホでツールチップが消えない

- **発生状況：** iPhone SE3でグラフにタッチすると黒い吹き出しが出て消えない
- **原因：** `chartCommon()` で `tooltip: { enabled: false }` を返していたが、各チャートの `plugins: { legend: {...} }` がオブジェクトごと上書き（シャローマージ）していた
- **解決：** 各チャートの `plugins` 内に直接 `tooltip: { enabled: !IS_MOBILE }` を記述。v2.1では `chartCommon()` 内に統合
- **代替案：** Chart.jsのグローバルデフォルト設定で無効化する方法もあったが、PC版では有効のままにしたいため個別制御を選択

### 課題3：ファイルがGitHub上で間違ったディレクトリに配置

- **発生状況：** ユーザーがPythonスクリプトをGitHubのルートに直接アップロード。本来は `muroto_offshore_current/` サブフォルダに入るべきだった
- **原因：** GitHub WebUIでのアップロード手順の理解不足
- **解決：** スクリプト側を修正し、フラット構造（ルート直下）でも動作するよう `update_offshore_dashboard_data.py` と `update_data.yml` のパスを調整
- **代替案：** ユーザーにファイル移動を依頼する案もあったが、GitHub WebUIでのフォルダ移動は煩雑なため、コード側を適応させた

### 課題4：測定点座標の誤り

- **発生状況：** 開発後に5地点すべての座標が意図した位置と異なることが発覚。7,815行すべてが無効データ
- **原因：** v1からv2への移行時に座標を再設計した際、正しい座標との照合が不十分だった
- **解決：** `config.py` / `index.html` の座標を修正。GitHub Actionsの `update_data.yml` に `collect_all` オプションを追加し、全期間データをGitHub Actions上で再収集（2時間48分）
- **代替案：** ローカルPCでの再収集も可能だったが、CMEMS認証がGitHub Secretsに設定済みであり、ローカル環境の構築不要なGitHub Actionsが確実だった

### 課題5：GitHub Actions 6時間制限への対応

- **懸念：** 2022年〜現在（約1,563日）の全データ再収集がGitHub Actionsの6時間制限内に収まるか
- **結果：** 実測2時間48分で完了。1日あたり約6.4秒の処理時間
- **判断：** 年次分割などの対策は不要と判断

## 設計思想

### 重視したポイント

1. **ゼロ依存デプロイ：** ビルドツール・フレームワーク不要。HTML+JS+CSSの1ファイルでダッシュボードが完結する。GitHub Pagesにそのまま配置可能
2. **初心者による保守可能性：** git CLIを使わず、GitHubのWebブラウザUIだけで全運用（ファイル更新・Actions実行・Secrets管理）が可能
3. **オフライン動作：** `file://` プロトコルでローカルHTMLを開くだけでダッシュボードが動作する（データJSファイル埋め込み方式）
4. **スマートフォンファースト：** 実際の利用シーンは釣り前の確認。スマホでの視認性・操作性を優先
5. **データ完全性：** 欠損チェッカーを内蔵し、データの連続性を常時可視化

### トレードオフ

| 選択 | 得たもの | 失ったもの |
|------|---------|----------|
| 静的HTML（フレームワーク不使用） | ゼロ依存・高速表示 | コンポーネント再利用性 |
| CSVテキスト埋め込みJS | `file://` 対応・CORS回避 | データファイルサイズ（630KB） |
| GitHub Actions毎日実行 | PC不要の自動運用 | GitHub無料枠の消費（月約50分） |
| フラットファイル構造 | GitHub運用の単純化 | ローカルとGitHubでのディレクトリ構造の乖離 |
| Chart.js（固定ライブラリ） | 安定性・CDN配信 | 最新チャートライブラリの機能 |

## 改善点・反省

### 技術的負債

1. **ローカルとGitHubのディレクトリ構造の乖離：** ローカルでは `muroto_offshore_current/` サブフォルダ内にPythonスクリプトがあるが、GitHubではルート直下。`run_full_data_collection.bat` の `cd muroto_offshore_current` と GitHub Actionsのルート実行の間に齟齬がある
2. **座標のハードコード：** `config.py` と `index.html` の座標バーで同じ座標を二重管理している。片方を変更した場合にもう片方を忘れるリスクがある（実際に座標修正時に両方の更新が必要だった）
3. **データJS埋め込み方式のスケーラビリティ：** CSVが増えるとJSファイルも比例して増大する。現在630KBだが、10年分になると約4MBに達する見込み

### 改善余地

1. `index.html` の座標バーを `config.py` から自動生成する仕組み（`update_offshore_dashboard_data.py` で座標情報もJS内に埋め込む）
2. 大規模データ対応として、CSVをJSON APIに変換しページネーション対応するアプローチ
3. PWA（Progressive Web App）化によるオフラインキャッシュ・ホーム画面追加
4. データの異常値検知（流速が通常範囲を大きく超えた場合のアラート）

## 再開発するなら

### 変更すべき点

1. **初期段階で座標を正確に確定する：** CMEMSデータのグリッド座標をプログラムで確認し、実際のグリッドポイントと目標座標の差分を可視化してから測定点を決定する
2. **ディレクトリ構造をGitHubとローカルで統一する：** 最初からフラット構造で設計するか、GitHub Actionsの `working-directory` 設定で対応する
3. **座標の一元管理：** `config.py` を単一の真実源（Single Source of Truth）とし、HTMLの座標表示は動的生成にする
4. **テスト自動化：** データ取得の成功判定・ダッシュボード表示の自動スクリーンショット比較をCI/CDに組み込む

### 継続すべき点

1. **静的HTML + CDNライブラリの構成：** ビルド不要・デプロイ即反映のシンプルさは大きな利点。この規模のプロジェクトではフレームワーク導入のメリットがない
2. **GitHub Actions + GitHub Pagesの組み合わせ：** 完全無料でサーバーレス運用が実現できている。運用コスト0円
3. **CSVを中間フォーマットとする設計：** Excel・Python・他ツールからも参照可能で、データの再利用性が高い
4. **欠損チェッカーの内蔵：** データ品質の可視化は運用開始後に価値が増す機能
5. **段階的な開発アプローチ（Phase分割）：** 動くものを早期に公開し、機能改善を重ねるプロセスはこの規模に適していた
